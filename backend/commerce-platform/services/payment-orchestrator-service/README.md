# Payment Orchestrator Service

Central payment entry point for checkout, refund, and provider webhook handling.

## Responsibilities

- Accept checkout requests and create orchestrator-managed payments.
- Dispatch payment creation/refund to provider adapters (Stripe/PayPal and others).
- Process provider webhooks and transition payment state machine safely.
- Publish commerce payment events to Kafka for downstream projection services.
- Enforce idempotency for checkout/refund requests.

## Main APIs

### Checkout

```http
POST /api/v1/payments/checkout
Idempotency-Key: <key>
```

Body fields:

- `userId`
- `userEmail`
- `provider` (`STRIPE`, `PAYPAL`, ...)
- `amount`
- `currency`
- `purpose`
- `metadata` (optional)

### Refund

```http
POST /api/v1/payments/refunds
Idempotency-Key: <key>
```

Body fields:

- `orchestratorPaymentId`
- `provider`
- `providerPaymentId`
- `reason` (optional)

### Webhook ingest

```http
POST /api/v1/webhooks/{provider}
```

Provider examples: `stripe`, `paypal`.

### Stripe: trigger webhook for the same request

When testing manually, Stripe webhook envelopes are generated by Stripe, but you can still map them to your exact checkout request:

- `providerPaymentId` from checkout response is the Stripe PaymentIntent id (`pi_...`).
- Metadata is propagated into Stripe objects/events (`userId`, `purpose`, and request metadata fields).

Local test loop:

```powershell
stripe listen --forward-to http://localhost:8999/api/v1/webhooks/stripe
```

```powershell
stripe payment_intents confirm <pi_id_from_checkout_response> --payment-method pm_card_visa
```

This emits `payment_intent.succeeded` for the same PaymentIntent created from your gateway/orchestrator request.

## Local configuration

Important keys in `application.properties`:

- `server.port=8999`
- `spring.datasource.*` (MySQL)
- `spring.data.redis.*` (idempotency cache)
- `spring.kafka.bootstrap-servers`
- `app.kafka.topic.payment-events=commerce.payment.events.v1`
- `app.payment.stripe.*`
- `app.payment.paypal.*`

## Event output

Publishes events to:

- `commerce.payment.events.v1`

Outbox relay behavior:

- Domain events are persisted to `outbox_events` first.
- Scheduled relay publishes pending rows to Kafka.
- Failed publish attempts remain retryable with backoff.

Typical event families:

- `PaymentInitiatedV1`
- `PaymentSucceededV1`
- `PaymentFailedV1`
- `PaymentRefundedV1`
- `PaymentStatusUpdatedV1`
