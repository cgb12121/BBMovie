WebSocket vs RSocket (how they differ)

WebSocket here: browser opens ws://…/api/watch-history/v1/ws?token=JWT; each message is a plain JSON string (same shape as POST /playback). No built-in routing or backpressure—simple and universal.

RSocket: separate TCP port on the watch-history service (default 8097, env RSOCKET_SERVER_PORT). Route playback.track, request–response with JSON payload; JWT is sent as Bearer token metadata at setup (Spring BearerTokenMetadata + RSocketRequester). Browsers do not speak raw TCP—use a desktop/CLI client, another microservice, or RSocket-over-WebSocket if you add that transport later.

Pause / seek / tab hidden flush immediately. Tab close uses fetch(..., keepalive: true) (JWT cannot use sendBeacon headers).
